# Sample arc — Maren Halvorson

> _Fictional. Maren is not a real person. Pine County, the IMLS-funded coalition, the metrics, the peer quotes — all invented for this example._

## The character (so you have context when reading)

**Maren Halvorson** — Director of Digital Services, Pine County Library System (rural county, ~110,000 residents across 7 branches).

**Career arc:**

| Years | Role | Scope |
|---|---|---|
| 2009 – 2012 | Reference Librarian, Maple Branch | First role post-MLS; built early adult programming |
| 2012 – 2017 | Branch Manager, Pine Falls Branch | First people-management role; 6 staff |
| 2017 – 2021 | Adult Services Coordinator (system level) | Cross-branch programming; founded the Digital Bridge curriculum |
| 2021 – present | Director of Digital Services | Executive team; led the IMLS grant + the mobile-hotspot lending program + fine elimination |

**What makes Maren's story interesting** (for the example's purposes):

- **Public-sector pay-and-power constraints.** Maren never had the discretionary budget of a tech leader — every program had to be funded externally or absorbed into a stretched operating budget. This forces a particular kind of leadership material: coalition-building, grant writing, default-design (because she couldn't pay her way out of friction).
- **Quantifiable outcomes despite a "soft" industry.** Public libraries get accused of being un-measurable. Maren's portfolio is the counter-example: program completion rates, grant amounts, retention deltas, hotspot loan volumes.
- **A real reversal in her own thinking.** The fine-elimination story (2022) was a position she'd argued *against* in 2017. The Reflection section in `stories/eliminating-overdue-fines.md` shows how a database captures a candidate's evolution honestly, not as a hero arc.

## Files in this example

```
sample-arc/
├── README.md                              (this file)
├── strengths.md                           (3 strengths — half the canonical 6, to keep the example tight)
├── evidence/
│   ├── digital-bridge-program.md          (multi-year program with completion-rate data)
│   └── imls-grant-coalition.md            (cross-org grant win)
├── stories/
│   └── eliminating-overdue-fines.md       (STAR story with Reflection beat)
├── voice/
│   └── peer-quotes.md                     (3 quotes with provenance)
├── SESSION.md                             (one session of handoff notes)
├── OPEN_QUESTIONS.md                      (tiering + one closed / one open item)
└── DECISIONS.md                           (one structural decision logged)
```

The full template (`../../template/`) has ~25 files. This example shows the *shape* of the load-bearing ones; the rest follow the same patterns.

## What this example does NOT cover

- **The `applications/<opening>/` per-opening variant work.** That's a separate workflow shown in `docs/WORKFLOWS.md`.
- **The `targets/` directory** (companies, recruiter personas, role criteria). Worth populating before any active job search; not shown here.
- **The `history/` directory.** Maren has 4 roles; in a real database each would get a `history/{role-slug}.md`. We're showing evidence + stories instead because those are the load-bearing layers for artifact generation.
- **A `CLAUDE.md` at the database root.** Mostly project-instructions, varies by user; see `template/CLAUDE.md` for the structure.

If you're a new user, read in this order: `README.md` (here) → `strengths.md` → one evidence file → the story → `SESSION.md` → `OPEN_QUESTIONS.md` → `DECISIONS.md`. About 15 minutes to internalize the rhythm.
